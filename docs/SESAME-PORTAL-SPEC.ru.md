# SesamePortal: техническое задание

## Назначение

`SesamePortal` - локально устанавливаемый web-портал видеонаблюдения для
управления пользователями, группами, камерами и несколькими серверами
SesameDVR.

Портал должен предоставлять:

- административный интерфейс управления системой;
- пользовательский интерфейс просмотра доступных камер;
- auth-backend для серверов SesameDVR;
- мозаичный просмотр камер;
- просмотр камер на карте OpenStreetMap;
- мобильную версию интерфейса и удобный мобильный video playback.

## Технологические требования

- Backend: PHP `8.2+`.
- Web server: nginx + php-fpm.
- База данных: SQLite по умолчанию.
- Архитектура хранения данных должна позволять в будущем перейти на
  MySQL/PostgreSQL.
- Frontend: server-rendered PHP templates или лёгкий JavaScript без обязательной
  SPA-архитектуры.
- Карта: OpenStreetMap через Leaflet.
- SSL: `certbot`.
- Визуальный стиль: логотип, шрифты, цвета и общий стиль взять от SesameDVR.

## Установка

Портал должен устанавливаться локально на сервер через install script.

Инсталлятор должен:

- принимать желаемый домен;
- принимать email администратора для выпуска ACME/Let's Encrypt сертификата;
- устанавливать системные зависимости: nginx, php-fpm, PHP extensions, SQLite,
  certbot;
- разворачивать приложение в `/opt/sesame-portal/current`;
- создавать state/config директорию `/var/lib/sesame-portal`;
- создавать nginx site-файл по имени домена или фиксированному имени
  `/etc/nginx/sites-available/sesame-portal.conf`;
- включать site через symlink в `/etc/nginx/sites-enabled`;
- настраивать php-fpm pool для портала;
- выпускать SSL-сертификат через certbot;
- создавать первого администратора;
- не перезаписывать чужие nginx site-файлы;
- поддерживать повторный запуск для repair/update.

Если nginx уже установлен и содержит другие сайты, инсталлятор должен создавать
отдельный site для SesamePortal и не менять чужие конфиги.

## Роли пользователей

В системе должно быть два набора привилегий:

- `admin` - полный доступ ко всем параметрам системы;
- `user` - только просмотр доступных камер.

Администратор может:

- менять все настройки системы;
- управлять серверами SesameDVR;
- управлять пользователями;
- управлять группами;
- управлять камерами;
- назначать пользователей и камеры в группы;
- видеть все камеры и все группы.

Обычный пользователь может:

- смотреть только камеры из групп, в которых состоит;
- фильтровать камеры по своим группам;
- добавлять доступные камеры в избранное;
- удалять камеры из избранного.

## Административная часть

Административный интерфейс должен включать разделы:

- Dashboard;
- Серверы SesameDVR;
- Пользователи;
- Группы;
- Камеры;
- Настройки;
- Журнал действий.

Все административные действия должны логироваться в audit log.

## Серверы SesameDVR

В настройках должна быть возможность добавлять несколько серверов SesameDVR.

Для каждого сервера задаются:

- название;
- URL сервера, например `https://dvr-1.example.com`;
- management key/token для управления;
- статус: active/blocked;
- дата последней проверки;
- результат последней проверки соединения.

Интерфейс должен уметь:

- добавлять сервер;
- изменять сервер;
- удалять сервер;
- блокировать/разблокировать сервер;
- проверять соединение с сервером.

Management key/token должен храниться в базе в защищённом виде.

## Пользователи

Интерфейс должен уметь:

- добавлять пользователей;
- удалять пользователей;
- изменять пользователей;
- блокировать пользователей;
- разблокировать пользователей;
- менять пароль;
- назначать роль `admin` или `user`.

Поля пользователя:

- login;
- password hash;
- роль;
- статус active/blocked;
- daily token;
- previous daily token;
- static integration token, опционально;
- дата создания;
- дата последнего входа.

Требования к паролю:

- минимум 6 символов;
- без ограничений на используемые символы;
- хранить только hash, например через `password_hash()` с `PASSWORD_DEFAULT`.

## Группы

Интерфейс должен уметь:

- добавлять группы;
- удалять группы;
- изменять группы;
- блокировать группы;
- разблокировать группы;
- добавлять пользователей в группы;
- удалять пользователей из групп;
- добавлять камеры в группы;
- удалять камеры из групп.

Поля группы:

- название;
- описание;
- статус active/blocked.

Если группа заблокирована, камеры этой группы не должны быть доступны обычным
пользователям через портал и auth-backend.

## Камеры

Интерфейс должен уметь:

- добавлять камеры;
- удалять камеры;
- изменять камеры;
- блокировать камеры;
- разблокировать камеры.

Поля камеры:

- имя камеры;
- URL-источник, например RTSP URL;
- сервер SesameDVR, на который камера должна писаться;
- режим выбора сервера: конкретный сервер или автоматический случайный выбор;
- положение на карте OpenStreetMap: latitude/longitude;
- направление обзора в градусах `0-359`;
- угол сектора обзора, опционально;
- глубина архива;
- статус active/blocked;
- технический ID/имя потока в SesameDVR.

При создании или изменении камеры портал должен через management API SesameDVR
создавать или обновлять соответствующий поток на выбранном DVR-сервере.

Если выбран автоматический случайный выбор сервера, портал должен выбрать один
из активных серверов SesameDVR и сохранить этот выбор в настройках камеры.

## Связь камер и групп

Камеры должны добавляться в одну или несколько групп.

Интерфейс должен уметь:

- добавлять камеру в группу;
- удалять камеру из группы;
- показывать список групп, где находится камера;
- показывать список камер внутри группы.

Обычный пользователь получает доступ к камере только если:

- пользователь active;
- камера active;
- хотя бы одна группа пользователя active;
- камера входит хотя бы в одну active-группу пользователя.

## Избранное

Для пользователей должен быть реализован персональный список избранных камер.

Функциональность:

- пользователь может добавить доступную ему камеру в избранное;
- пользователь может удалить камеру из избранного;
- избранное хранится отдельно для каждого пользователя;
- администратор также может иметь своё личное избранное;
- пользователь не может добавить в избранное камеру, к которой у него нет
  доступа;
- если камера стала недоступна пользователю из-за изменения групп, блокировки
  камеры или блокировки группы, она не должна отображаться в избранном.

Фильтры для обычного пользователя:

- `Все мои группы`;
- `Избранное`;
- конкретные группы пользователя.

Фильтры для администратора:

- `Все камеры`;
- `Избранное`;
- любая группа системы.

Фильтр `Избранное` должен работать и в мозаике, и на карте.

## Просмотр камер

Портал должен предоставлять два основных режима просмотра.

### Мозаика

Мозаика должна отображать доступные пользователю камеры.

Для каждой камеры:

- preview/thumbnail;
- название;
- статус доступности, если он известен;
- кнопка добавления/удаления из избранного;
- переход в embed player SesameDVR.

Мозаика должна поддерживать фильтры:

- по всем доступным группам пользователя;
- по конкретной группе;
- по избранному.

### Карта OpenStreetMap

Карта должна отображать доступные пользователю камеры на OSM.

Для каждой камеры:

- маркер на карте;
- направление обзора;
- сектор обзора, если задан;
- название;
- кнопка добавления/удаления из избранного;
- переход в embed player SesameDVR.

Карта должна поддерживать те же фильтры, что и мозаика.

## Embed player

Просмотр камеры должен открываться через embed player SesameDVR.

Портал должен формировать ссылку на embed player с пользовательским token,
который проходит через auth-backend SesamePortal.

Обычный пользователь не должен получать прямой доступ к management API
SesameDVR.

## Мобильная версия

Необходимо реализовать адаптивную мобильную верстку для всего интерфейса
портала.

Требования:

- интерфейс должен корректно работать на смартфонах и планшетах;
- основные сценарии должны быть удобны с touch-управлением;
- меню, фильтры, список камер и карта должны адаптироваться под малый экран;
- мозаика камер на мобильном должна переходить в 1-2 колонки в зависимости от
  ширины экрана;
- элементы управления должны быть достаточно крупными для нажатия пальцем;
- карта OSM должна поддерживать удобное масштабирование и выбор камеры на
  touch-устройстве;
- embed/video player должен быть удобен на мобильном устройстве;
- при открытии камеры на мобильном устройстве приоритет должен быть у видео,
  без лишней административной обвязки;
- fullscreen playback должен работать там, где это поддерживается браузером;
- интерфейс не должен требовать горизонтального скролла.

## Auth-backend для SesameDVR

SesamePortal должен предоставлять auth-backend endpoint для серверов SesameDVR.

Пример endpoint:

```text
GET /api/sesamedvr/auth
```

Endpoint должен принимать параметры auth-запроса от SesameDVR и проверять:

- существует ли пользователь по переданному token;
- не заблокирован ли пользователь;
- не заблокирована ли камера;
- не заблокированы ли группы, через которые пользователь получает доступ;
- имеет ли пользователь право смотреть запрошенную камеру.

Ответ должен быть совместим с ожиданиями SesameDVR auth-backend flow:

- разрешить playback;
- запретить playback.

## Пользовательские токены

У каждого пользователя должен быть персональный daily token.

Правила daily token:

- новый token генерируется раз в сутки;
- предыдущий daily token остаётся валиден ещё 6 часов;
- с `00:00` до `06:00` каждых суток валидны и старый, и новый token;
- после `06:00` валиден только новый token;
- ротация token должна выполняться cron-задачей или встроенным scheduler.

Также должна быть возможность сгенерировать дополнительный постоянный static
integration token для конкретного пользователя.

Правила static token:

- static token не ротируется автоматически;
- static token можно отозвать;
- static token можно выпустить заново;
- static token должен быть помечен в UI отдельно от daily token;
- static token должен показываться пользователю или администратору только при
  генерации.

Заблокированный пользователь не должен проходить auth-backend ни по daily token,
ни по previous daily token, ни по static token.

## Минимальные backend/API функции

Портал должен реализовать:

- login/logout;
- CRUD пользователей;
- CRUD групп;
- CRUD камер;
- CRUD серверов SesameDVR;
- управление membership пользователь-группа;
- управление membership камера-группа;
- управление избранными камерами пользователя;
- auth-backend endpoint для SesameDVR;
- token rotation;
- выпуск и отзыв static token;
- список камер для мозаики;
- список камер для карты;
- фильтрацию по группам и избранному.

## Безопасность

Требования:

- пароли хранить только в hash;
- management keys SesameDVR хранить защищённо;
- admin routes защищать session auth;
- реализовать CSRF-защиту для форм;
- вести audit log действий администратора;
- static integration token показывать только при генерации;
- заблокированный пользователь не должен проходить auth-backend;
- заблокированная камера не должна быть доступна для просмотра;
- заблокированная группа не должна давать доступ к камерам;
- обычный пользователь не должен видеть камеры вне своих active-групп.

## Критерии приёмки

- Портал устанавливается одной install-командой.
- nginx site создаётся под указанный домен.
- HTTPS-сертификат выпускается через certbot.
- Инсталлятор не ломает существующие nginx site-файлы.
- Первый admin-пользователь создаётся при установке.
- Админ может добавить SesameDVR сервер и проверить соединение.
- Админ может управлять пользователями, группами и камерами.
- Админ может добавлять пользователей и камеры в группы.
- Камера создаётся или обновляется на выбранном SesameDVR сервере.
- Обычный пользователь видит только камеры своих active-групп.
- Админ видит все камеры и все группы.
- Мозаика показывает камеры с preview и открывает embed player.
- Карта OSM показывает камеры, направления обзора и открывает embed player.
- Интерфейс удобно работает на мобильном телефоне.
- Мозаика, карта и player адаптируются под мобильный экран.
- Пользователь может добавить и удалить камеру из избранного.
- Фильтр `Избранное` работает в мозаике и на карте.
- Избранное не показывает камеры, к которым пользователь больше не имеет
  доступа.
- Auth-backend разрешает доступ только по актуальному daily/static token.
- В период `00:00-06:00` валидны старый и новый daily token.
- После `06:00` старый daily token перестаёт быть валидным.
- Заблокированные пользователи, группы и камеры не дают доступ к playback.
