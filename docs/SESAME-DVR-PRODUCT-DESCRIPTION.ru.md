# Sesame DVR: описание продукта

Sesame DVR — это серверная система видеозаписи и просмотра IP-камер. Продукт
устанавливается на Linux-сервер, подключается к камерам и другим видеоисточникам
по RTSP/HTTP URL и UDP multicast, принимает push-доставку по RTMP/SRT, ведёт
архив, отдаёт live-видео и предоставляет web-интерфейс для управления потоками,
просмотра архива и диагностики состояния системы.

Sesame DVR можно использовать как лёгкий DVR/NVR-сервер для объектов, где нужно
централизованно принимать видеопотоки, хранить записи, быстро находить нужный
фрагмент по времени и предоставлять доступ к live или архивному видео через
браузер, API или встраиваемый плеер.

## Для чего используется

- запись архива с IP-камер, RTSP/HTTP-источников, UDP multicast и push-потоков;
- SRT/MPEG-TS доставка от Edge Agent или внешнего publisher-а в SesameDVR;
- удалённый просмотр live-видео через браузер;
- просмотр архива по временной шкале;
- ускоренный просмотр длительных периодов через timelapse-архив;
- выгрузка нужного фрагмента архива в MP4;
- публикация камер и архива для клиентов, операторов или внешних систем;
- встраивание видеоплеера в сторонние web-приложения;
- мониторинг состояния камер, ingest-процессов, сервера и подключённых клиентов;
- публикация OpenMetrics/Prometheus метрик для внешнего мониторинга и alerting;
- сбор ONVIF motion events и сопоставление событий движения с видеоархивом;
- хранение архива в SingleVolume или MultiVolume режиме с per-volume каталогами;
- централизованное управление большим количеством камер в рамках лимитов лицензии
  и мощности сервера.

## Основные возможности

### Подключение и запись камер

Sesame DVR принимает видеопотоки от IP-камер и других источников, доступных по
RTSP или совместимым HTTP URL, локальные UDP multicast/MPEG-TS потоки, а также
push-потоки через RTMP/FLV или SRT/MPEG-TS. Для каждой камеры можно задать имя,
источник, параметры записи, срок хранения архива, режим авторизации
playback-доступа и дополнительные настройки ingest.

Для UDP multicast источников можно указать `sourceType=udp_multicast` и source
вида `udp://@239.10.10.10:5000?localaddr=192.168.0.1...`, чтобы явно выбрать
сетевой интерфейс подписки на multicast group.

Для объектов за NAT или в нестабильных сетях можно использовать SesameDVR Edge
Agent: агент читает локальную камеру по RTSP/ONVIF и доставляет поток на
центральный DVR по SRT/MPEG-TS. SesameDVR принимает такие камеры как обычные
`sourceType=push` потоки, пишет архив и отдаёт live/архив через HLS, WebRTC и
embed-плеер. SRT работает через общий UDP-порт с маршрутизацией по
`streamid=sesame:stream=<streamName>`, поэтому отдельный публичный порт на
каждую камеру не требуется.

Сервис автоматически запускает включённые камеры при старте, следит за их
runtime-состоянием и может перезапускать поток, если запись архива перестала
обновляться. В protected-версии основной ingest path — native `ffmpeg_nif`,
который пишет self-initializing fMP4 segment files сразу в выбранный storage
volume без отдельного OS-процесса `ffmpeg` на каждый совместимый поток.

### Live-просмотр

Для просмотра в браузере доступны:

- live HLS-потоки;
- native WebRTC/WHEP для H.264 и поддерживаемого браузером HEVC;
- optional G.711 audio passthrough для совместимых потоков;
- fallback на HLS, если WebRTC для конкретного потока или браузера недоступен.

Это позволяет использовать Sesame DVR как источник live-видео для операторской
панели, клиентского портала или отдельной страницы просмотра камеры.

### Видеоархив

Sesame DVR пишет durable DVR-архив на диск в виде самодостаточных fMP4
сегментов и индексирует их для быстрого доступа к нужному диапазону времени.
Архив можно смотреть через HLS-плейлист по заданному временному диапазону,
открывать в web-интерфейсе или использовать из стороннего приложения через HTTP
endpoints.

Для камер с ONVIF-событиями embed-плеер умеет воспроизводить архив в режиме
только движения: участки без motion events пропускаются, а таймлайн остаётся
привязанным к реальному времени архива.

Для камер, где включён timelapse, SesameDVR сохраняет отдельный ускоренный
видеоархив. Сырые timelapse-кадры используются как staging, затем фоновая
задача собирает из них HLS/fMP4 chunks и обновляет готовый timelapse manifest.
Поэтому просмотр timelapse не требует каждый раз строить playlist обходом
архива. В плеере timelapse имеет отдельную длительность воспроизведения, но
timeline и подпись текущего кадра остаются привязаны к реальному времени
архива.

Для каждой камеры можно задать срок хранения. Старые записи очищаются
автоматически по retention-настройкам и ограничениям лицензии.

### Хранилище и масштабирование

Для обычной установки используется `SingleVolume`: default архив находится в
`/var/dvr/segments`. Для крупных инсталляций доступен `MultiVolume`: несколько
томов хранения могут быть online одновременно, а новые сегменты распределяются
по политике `single`, `round_robin`, `weighted_round_robin` или `least_used`.

Каждый volume имеет собственный catalog и hour indexes. Если один volume стал
offline, archive playback показывает gaps только на диапазонах этого volume, а
остальные тома остаются доступными. Публичные archive URLs для MultiVolume имеют
вид `/dvr/v/<volume_id>/<camera>/...`; managed nginx site отдаёт такие сегменты
напрямую с диска и обновляется штатным updater'ом.

Для быстрых запросов к архиву используются три уровня индексов: `HourIndex`,
`CameraIndex` и `VolumeIndex`. В актуальной protected-версии основные форматы
хранения индексов - `*.hidx` для часовых индексов и `*.cidx` для индекса
камеры. Они рассчитаны на дешёвое append/patch-обновление вместо полной
перезаписи больших `.term` snapshots. Обновления распространяются асинхронно:
запись сегмента обновляет `HourIndex`, затем materializer обновляет
`CameraIndex`, а после этого `VolumeIndex`.

Запись сегментов, commit, обновление индексов и физическое удаление проходят
через общий Native IO / Disk IO scheduler. Это позволяет задавать приоритеты и
лимиты на IO по классам работ, чтобы архивная запись оставалась приоритетной
даже во время retention, rebuild или удаления больших объёмов данных.

### Выгрузка фрагментов

Пользователь может выбрать нужный участок архива и получить MP4-файл. Это
удобно для передачи записи клиенту, службе безопасности, технической поддержке
или для сохранения доказательного фрагмента события.

### Web-интерфейс администратора

Встроенная admin-панель позволяет управлять системой без ручного редактирования
конфигов. Через UI можно:

- добавлять, изменять, запускать, останавливать, включать и отключать потоки;
- смотреть состояние всех камер и архива;
- видеть, какие потоки пишут архив, какие запущены, какие остановлены или имеют
  ошибки;
- открывать live и archive playback;
- смотреть логи выбранного потока и глобальный runtime-лог;
- запускать ffprobe-диагностику по камере;
- управлять playback-авторизацией;
- смотреть активных playback client users, активные WebRTC-клиенты и недавние
  HLS-клиенты;
- работать с ONVIF-устройствами и событиями движения;
- управлять storage volumes, audit и rebuild catalog;
- смотреть dashboard состояния сервера.

При первом входе используется временный пароль `admin / admin`; интерфейс должен
потребовать сменить пароль администратора.

### ONVIF и события движения

Sesame DVR умеет сканировать ONVIF-устройства на основе RTSP-источников,
проверять доступность устройств, сохранять список ONVIF-камер и собирать
PullPoint events. ONVIF-устройство можно добавить из выбранного RTSP-потока:
система попробует найти endpoint именно для этой камеры и попросит подтвердить
добавление перед записью в конфиг.

События движения сохраняются отдельно от видеоархива и отображаются в UI на
общей временной шкале вместе с доступными интервалами архива. Это помогает
быстрее находить моменты, где камера зафиксировала движение.

### Встраиваемый плеер и API

Для каждой камеры доступен embed-плеер, который можно встроить в сторонний
интерфейс. Он поддерживает live-видео, DVR-таймлайн, переход к архивному
времени, воспроизведение архива по событиям движения и выгрузку MP4-фрагмента.

Также доступны HTTP endpoints для live HLS, архивных HLS-плейлистов, MP4 export,
preview, статуса записи, WebRTC/WHEP и management API. Это позволяет
интегрировать Sesame DVR с клиентскими порталами, внутренними системами
мониторинга, биллингом, CRM или отраслевыми приложениями.

### Мониторинг и диагностика

Dashboard показывает текущее состояние сервера и историю метрик. В интерфейсе
доступны данные по CPU, памяти, дискам, сети, ingest-нагрузке, состоянию потоков,
процессам, клиентским сессиям и логам.

Для внешнего мониторинга доступен endpoint `/metrics` в OpenMetrics формате.
Авторизация endpoint настраивается отдельно от admin UI и playback tokens:
`disabled`, `none`, `bearer` или `basic`. Это позволяет подключать Prometheus,
Grafana и alert rules без выдачи management token сторонней системе.

Для крупных инсталляций Dashboard также показывает состояние Native IO Gateway,
RawFileIO, Disk IO pressure, DeleteQueue, materializer backlog, lag
материализации индексов, cache hit-rate и BEAM scheduler pools. Дорогие
диагностические секции открываются вручную через debug snapshot, чтобы обычное
обновление Dashboard не создавало дополнительную нагрузку на production-сервер.

Это помогает администратору быстро понять, связана ли проблема с камерой,
сетевым доступом, диском, CPU, ffmpeg/ingest или общей нагрузкой на сервер.

### Публичная публикация

Инсталлятор может опубликовать Sesame DVR наружу через nginx. Приложение
остаётся на внутреннем `127.0.0.1:3000`, а nginx принимает внешние HTTP/HTTPS
запросы, проксирует web-интерфейс и API, а DVR-сегменты отдаёт напрямую с диска.

При указании домена инсталлятор может выпустить Let's Encrypt сертификат через
ACME и включить HTTPS. Если на сервере уже есть собственный nginx-конфиг,
инсталлятор не должен перезаписывать чужие сайты без явного домена для Sesame
DVR.

### Лицензирование и обновления

Protected-установка активируется лицензионным ключом через license server.
Лицензия привязывается к серверу и определяет доступные лимиты, например
количество камер, срок действия, максимальный retention и максимальное число
активных playback client users. Playback client user считается по сочетанию
IP-адреса клиента и `User-Agent`; лимит применяется к HLS и WebRTC-доступу.

Для обновления установленной версии используется штатная команда:

```bash
sudo sesame-dvr-update
```

Обновление скачивает опубликованный protected artifact и применяет его без
локальной сборки проекта на сервере клиента.

Для узкого класса BEAM-only исправлений SesameDVR может публиковать отдельный
`beam_hot_patch` artifact. Он применяется без restart `sesame-dvr.service`,
только для заранее разрешённых модулей и только если текущий `buildId` совпадает
с `fromBuildId` manifest. Полное release-обновление остаётся основным и
обязательным путём для NIF/native кода, frontend assets, systemd/installer
изменений, конфигурации запуска и state migrations.

## Типовой сценарий работы

1. Администратор подготавливает Linux-сервер и домен.
2. Запускает install-команду с лицензионным ключом.
3. Инсталлятор ставит зависимости, сервис, nginx-публикацию и активирует
   лицензию.
4. Администратор открывает admin UI, меняет пароль и добавляет камеры.
5. Sesame DVR начинает писать архив и отдавать live-видео.
6. Операторы или внешние системы используют web-интерфейс, embed-плеер или API
   для просмотра live, архива и выгрузки MP4.

## Где продукт особенно полезен

- объекты с набором IP-камер, где нужен централизованный видеоархив;
- интеграторы, которым нужно быстро развернуть DVR-сервер у клиента;
- клиентские порталы, где видео должно быть доступно из браузера;
- системы мониторинга объектов, строительных площадок, складов, офисов,
  производств и удалённых локаций;
- проекты, где важно иметь HTTP/API-доступ к live и архивному видео без
  тяжёлой VMS-платформы;
- инсталляции, где требуется контролируемое хранение архива с автоматической
  очисткой старых записей.

## Эксплуатационная модель

Sesame DVR работает как `systemd` service. Основные данные хранятся в файловой
системе:

- конфигурация и license state — в `/var/lib/sesame-dvr`;
- default видеоархив — в `/var/dvr/segments`;
- дополнительные архивные тома — в root'ах, заданных в storage volumes;
- preview-файлы — в `/var/dvr/previews`;
- ONVIF events — в `/var/dvr/onvif-events`;
- приложение — в `/opt/sesame-dvr/current`.

База данных для архива не требуется: источником истины остаётся файловая система
с видео-сегментами и primary `*.hidx` hour indexes. Производные `*.cidx`
CameraIndex и VolumeIndex можно восстановить по HourIndex, а устаревшие legacy
`.term` индексы читаются только для совместимости во время миграции.

## Важные ограничения

Sesame DVR не является системой видеоаналитики общего назначения. Он не делает
распознавание лиц, номеров, объектов или поведенческую аналитику без внешних
интеграций.

WebRTC-воспроизведение работает без транскодирования и зависит от codec support
конкретного потока и браузера. Для неподдерживаемых WebRTC-сценариев доступен
HLS playback.

Производительность и максимальное количество камер зависят от CPU, дисковой
подсистемы, сети, codec-профилей камер, bitrate, длительности хранения архива и
того, какие функции включены на конкретной инсталляции. Protected runtime
рассчитан на `ffmpeg_nif` ingest; внешний CLI `ffmpeg` producer остаётся для
unprotected/dev-сценариев и не является production fallback в protected-сборке.
